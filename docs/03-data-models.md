# Data Models

**Document:** 03-data-models.md  
**Purpose:** Define all data structures used in the application

---

## Overview

The app has two primary entities:

1. **Bean** – A coffee bean entry in the user's library
2. **Shot** – An espresso shot pulled using a specific bean

Relationship: One Bean has many Shots.

---

## Type Definitions

These types should be created BEFORE feature development begins. They are the contract between all parts of the codebase.

### Bean

```typescript
// src/types/bean.ts

/**
 * Represents a coffee bean in the user's library.
 * The primary object users interact with.
 */
interface Bean {
  /** Unique identifier (UUID) */
  id: string;
  
  /** Name of the coffee (e.g., "Ethiopia Yirgacheffe") */
  name: string;
  
  /** Name of the roaster (e.g., "Square Mile") */
  roaster: string;
  
  /** Date the coffee was roasted (ISO string, optional) */
  roastDate: string | null;
  
  /** User's personal rating (1-5 scale, optional) */
  rating: number | null;
  
  /** Free-form notes about this bean */
  notes: string;
  
  /** The recipe the user landed on when dialing in */
  dialedRecipe: DialedRecipe | null;
  
  /** Whether the bean has been marked as "dialed in" */
  isDialedIn: boolean;
  
  /** Timestamp when this bean was added to the library */
  createdAt: string;
  
  /** Timestamp when this bean was last modified */
  updatedAt: string;
}

/**
 * The final recipe saved when user marks a bean as "dialed".
 * Extracted from the winning shot.
 */
interface DialedRecipe {
  /** Dose in grams (e.g., 18) */
  doseGrams: number;
  
  /** Yield in grams (e.g., 36) */
  yieldGrams: number;
  
  /** Extraction time in seconds (e.g., 28) */
  timeSeconds: number;
  
  /** Grind setting (machine-specific, user-entered string) */
  grindSetting: string;
  
  /** Calculated ratio (yield / dose) */
  ratio: number;
  
  /** ID of the shot this recipe came from */
  sourcesShotId: string;
  
  /** When this recipe was saved */
  savedAt: string;
}
```

### Shot

```typescript
// src/types/shot.ts

/**
 * Represents a single espresso shot.
 * Always belongs to a Bean.
 */
interface Shot {
  /** Unique identifier (UUID) */
  id: string;
  
  /** ID of the bean this shot belongs to */
  beanId: string;
  
  /** Dose in grams */
  doseGrams: number;
  
  /** Yield in grams */
  yieldGrams: number;
  
  /** Extraction time in seconds */
  timeSeconds: number;
  
  /** Grind setting (optional, user-entered string) */
  grindSetting: string;
  
  /** Calculated ratio (yield / dose) */
  ratio: number;
  
  /** User's taste assessment */
  taste: TasteFeedback;
  
  /** Free-form notes about this specific shot */
  notes: string;
  
  /** Order within the dial-in session (1 = first shot, etc.) */
  shotNumber: number;
  
  /** Whether this shot was marked as the dialed-in result */
  isDialedShot: boolean;
  
  /** Timestamp when this shot was logged */
  createdAt: string;
}

/**
 * Taste feedback for a shot.
 * Simplified to a single-axis assessment for Phase 0.
 */
interface TasteFeedback {
  /**
   * Balance assessment on a scale from -2 to +2:
   * -2 = Very sour (under-extracted)
   * -1 = Slightly sour
   *  0 = Balanced
   * +1 = Slightly bitter
   * +2 = Very bitter (over-extracted)
   */
  balance: -2 | -1 | 0 | 1 | 2;
  
  /**
   * Optional: Body assessment
   * We may add this later but keeping it simple for Phase 0
   */
  // body?: 'thin' | 'medium' | 'heavy';
  
  /**
   * Optional: Strength assessment  
   * We may add this later but keeping it simple for Phase 0
   */
  // strength?: 'weak' | 'medium' | 'strong';
}
```

## Database Schema

Using Dexie.js, our IndexedDB schema:

```typescript
// src/db/database.ts

import Dexie, { Table } from 'dexie';
import { Bean, Shot } from '../types';

class EspressoDatabase extends Dexie {
  beans!: Table<Bean, string>;
  shots!: Table<Shot, string>;

  constructor() {
    super('EspressoDialIn');
    
    this.version(1).stores({
      // Primary key is 'id', indexed fields after
      beans: 'id, roaster, createdAt, updatedAt',
      shots: 'id, beanId, createdAt, isDialedShot'
    });
  }
}

export const db = new EspressoDatabase();
```

### Indexes Explained

**beans table:**
- `id` – Primary key, for direct lookups
- `roaster` – For filtering/grouping by roaster
- `createdAt` – For sorting by date added
- `updatedAt` – For sorting by recent activity

**shots table:**
- `id` – Primary key
- `beanId` – For fetching all shots for a bean (critical)
- `createdAt` – For chronological ordering
- `isDialedShot` – For finding the dialed shot quickly

## Validation Rules

### Bean Validation

| Field | Rule |
|-------|------|
| name | Required, 1-100 characters |
| roaster | Required, 1-100 characters |
| roastDate | Optional, valid ISO date string if provided |
| rating | Optional, integer 1-5 if provided |
| notes | Optional, max 1000 characters |

### Shot Validation

| Field | Rule |
|-------|------|
| beanId | Required, must reference existing bean |
| doseGrams | Required, number 1-30 |
| yieldGrams | Required, number 1-100 |
| timeSeconds | Required, number 1-120 |
| grindSetting | Optional, max 50 characters |
| taste.balance | Required, integer -2 to +2 |
| notes | Optional, max 500 characters |

### Calculated Fields

These are computed, not user-entered:

| Field | Calculation |
|-------|-------------|
| ratio | yieldGrams / doseGrams |
| shotNumber | Count of existing shots for bean + 1 |
| createdAt | Current timestamp on creation |
| updatedAt | Current timestamp on any modification |

## ID Generation

Use UUIDs for all entity IDs:

```typescript
// src/utils/id.ts

export function generateId(): string {
  return crypto.randomUUID();
}
```

This ensures:
- No collisions
- IDs are generated client-side (no server needed)
- Future-proof for sync scenarios

## Migration Strategy

For Phase 0, we start at version 1 with no migrations.

When we need to change the schema:

1. Increment version number
2. Define migration in Dexie's `upgrade()` method
3. Test migration with sample data
4. Document breaking changes

```typescript
// Example future migration
this.version(2).stores({
  beans: 'id, roaster, createdAt, updatedAt, isDialedIn',
  shots: 'id, beanId, createdAt, isDialedShot'
}).upgrade(tx => {
  // Migration logic here
  return tx.table('beans').toCollection().modify(bean => {
    // Transform old data to new format
  });
});
```

## Summary

| Entity | Primary Key | Key Relationships |
|--------|-------------|-------------------|
| Bean | id (UUID) | Has many Shots |
| Shot | id (UUID) | Belongs to one Bean (via beanId) |

These types are the foundation. All features must use these definitions—no ad-hoc object shapes allowed.
